Student Reading: Learn This First

Begin by defining the symptom. The symptom is what the circuit does or does not do. “LED off” is a symptom. “Bad LED” is a possible cause, not evidence.

Next define expected operation. You must know what should happen before you can explain what is wrong. Expected operation comes from the schematic, plan, or instructor description.

Use the troubleshooting order: observe symptom, make safe, inspect, compare to expected path, choose one test, collect evidence, correct one issue, verify, and document.

Do not guess. Replacing parts without evidence can hide the real problem and waste time. Changing several things at once makes it impossible to know what fixed the circuit.

Evidence examples include a visible loose wire, measured source voltage, open continuity path with power off, wrong resistor row, reversed LED, or switch terminal mismatch.

Use this repair-ticket sentence frame: “Symptom was __. Evidence showed __. Correction was __. Verification showed __.”

Lesson Work

  1. Sort statements as symptom, possible cause, evidence, correction, or verification.
  2. Practice the troubleshooting flow on short scenarios.
  3. Write an expected-vs-actual table for a simple circuit.
  4. Draft a repair ticket using evidence, not guessing.
